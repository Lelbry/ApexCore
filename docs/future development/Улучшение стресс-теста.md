## 1. Контекст и наблюдения

После фиксов Roofline-багов и логики вердикта (commit `12fcf8c`) на
i9-12900K + DDR5-6400 dual-channel пользователь видит:

- **DGEMM 96 GFLOPS = 8.3%** от Roofline-пика 1165 GFLOPS (P+E split fixed).
- **STREAM 15.25 GB/s = 14.9%** от Roofline-пика 102.4 GB/s.
- **CPU load 81%**, не 100%.

Литература говорит что **реально достижимо**:

- MKL/OpenBLAS DGEMM на x86 в **изоляции** — **>90% Roofline**
  ([Salykova 2024](https://salykova.github.io/matmul-cpu)). На гибридной
  Alder/Raptor Lake с правильным affinity OpenBLAS на P-cores даёт
  **~400 GFLOPS на i9-14900K** ([Intel Community](https://community.intel.com/t5/Intel-oneAPI-Math-Kernel-Library/Weird-DGEMM-with-i9-14900k/m-p/1692021)).
  Без affinity Hybrid-scheduling роняет throughput.
- STREAM Triad в изоляции — **80-95% от theoretical peak** с правильной
  affinity ([Lemire 2024](https://lemire.me/blog/2024/01/13/estimating-your-memory-bandwidth/),
  [Microsoft STREAM guide](https://learn.microsoft.com/en-us/azure/high-performance-computing/performance-benchmarking/stream-benchmark)).
  Single-thread init без NUMA-awareness — **до −50% на 2-socket**.

То есть наши 8% / 15% — это **в 4-5 раз меньше изоляционного потолка**.
В parallel-режиме (DGEMM + STREAM одновременно) потери ожидаемы, но не
в такой степени.

Из аудита кода (`builtin_large_dgemm.py`, `builtin_large_stream.py`,
`parallel_runner.py`):

- **Нет warmup-фазы** — cold cache + cold BLAS pthread-pool на старте.
- **Нет CPU affinity** — OS-scheduler мешает P-core и E-core threads.
- **Нет BLAS thread control** — OpenMP/pthreads BLAS видит все 24 logical
  CPU включая E-cores, тогда как DGEMM на гибриде выгоднее на P-cores only.
- **Нет diagnostic-режима** — пользователь видит финальное число без
  разбора. Невозможно отличить «BLAS не настроен» от «контентия с STREAM».

**Цель улучшения** — поднять реальную нагрузку до **30-50% Roofline для
DGEMM в изоляции** (и 15-25% в parallel), **60-80% Roofline для STREAM
в изоляции** (25-40% в parallel). Решение **универсальное**: hybrid-
специфичные оптимизации активируются только при детекции гибридного CPU,
на не-целевом железе — graceful fallback.

## 2. Phase 1 — Литературный обзор

**Цель:** сформулировать realistic expected ratios для DGEMM и STREAM на
разных классах x86 железа со ссылками на peer-reviewed литературу.

**Источники для проверки (через WebFetch):**

- **McCalpin 1995** «Memory Bandwidth and Machine Balance in Current High
  Performance Computers». IEEE TCCA. Канон STREAM Triad: workload ≥ 4×L3,
  per-thread init, no NUMA crossing.
- **Goto & van de Geijn 2008** «Anatomy of High-Performance Matrix
  Multiplication». ACM TOMS 34(3):12. Tile sizes (L1/L2/L3 blocking),
  packing overhead, achievable %.
- **Williams 2009** Roofline — operational intensity DGEMM (≈N/3 FLOP/byte
  для больших N — compute-bound) vs STREAM (1/12 FLOP/byte — memory-bound).
- **Intel Optimization Reference Manual** Alder Lake §2.4 — hybrid topology,
  AVX2 throughput на P-cores vs E-cores.
- **Salykova 2024** — empirical DGEMM > MKL на Zen 4 при правильных
  оптимизациях.
- **Bergstrom (UChicago TR-2011-02)** «Measuring NUMA effects with STREAM».
- **Phoronix / AnandTech / Chips and Cheese** — empirical 50-85 GB/s для
  DDR5-6400 desktop.

**Целевые числа после Phase 3:**

- DGEMM в изоляции, гибридный desktop Intel: **30-50% Roofline**.
- STREAM в изоляции, dual-channel DDR5 desktop: **60-80% Roofline**.
- Parallel-режим (contention): **на ~30-40% ниже** изоляции.

**Артефакт:** §1-2 финального `docs/research/stress_engines_analysis.md`.

## 3. Phase 2 — Diagnostic mode

**Цель:** дать инструмент для root cause analysis. Без него рекомендации —
догадки.

### 3.1 CLI команда `apexcore stress diagnose`

Новый файл `interfaces/cli/commands/stress_diagnose.py` + регистрация
подкоманды в `interfaces/cli/commands/stress.py`.

Что делает:
1. **Environment dump:** `numpy.show_config()`, `numpy.__version__`,
   обнаруженный BLAS (через `threadpoolctl.threadpool_info()` если есть,
   иначе парсинг numpy config), env-vars MKL/OPENBLAS/OMP thread,
   `os.cpu_count()`, hybrid topology через `_detect_hybrid_topology`.
2. **Isolation runs** (по 15 сек):
   - DGEMM-only (без STREAM).
   - STREAM-only (без DGEMM).
   - Parallel (DGEMM + STREAM, как сейчас).
3. **Таблица результатов:** для каждого режима — measured / roofline_peak / %.
4. **Анализ:** автоматический вывод «root cause hypothesis» — если
   DGEMM-only < 30% Roofline → подсказка «BLAS не настроен на P-cores»;
   если parallel падает в 2× от isolation → подсказка «memory contention
   значительная»; и т.п.

### 3.2 Isolation-режим в `ParallelStressRunner`

Файл: `application/parallel_runner.py`. Добавить `run_isolated(spec,
duration_sec, cancel_token)` — запуск одного движка через тот же путь что
`run()`, но без второго потока. Переиспользует существующую логику
(StressResult, error tracking) для diagnostic command. Не меняет parallel
API.

### 3.3 Тесты

`tests/unit/test_stress_diagnose.py` (новый):
- Моки `BuiltinLargeDgemmEngine.run` / `BuiltinLargeStreamEngine.run`
  возвращают фиксированные throughput.
- Проверка форматирования таблицы.
- Проверка анализа root cause.

## 4. Phase 3 — Архитектурные улучшения

Каждый sub-phase — изолированное изменение с измеримым эффектом. Все
улучшения **универсальны**: на не-целевом железе либо no-op, либо
non-вредительный fallback.

### 4.1 Phase 3a — Warmup phase

**Зачем.** На 30-секундном прогоне первые 1-2 сек тратятся на page-fault'ы,
прогрев L3, разогрев BLAS pthread-pool. Включаются в throughput → −5-15%.

**Файлы:**
- `infrastructure/stress/builtin_large_dgemm.py`: добавить
  `_warmup(a, b, c, duration_sec=1.5)` перед основным циклом и до
  `t0 = now()`.
- `infrastructure/stress/builtin_large_stream.py`: то же для per-thread
  `a, b, c`.
- `infrastructure/stress/base.py` `run_threaded_loop` — добавить
  `warmup_func` параметр, выполняется ВНЕ measurement window.

**Универсальность.** Работает на любом CPU. Если `duration_sec ≤ 5`
warmup не выполняется (нет смысла на коротком прогоне).

**Тесты.** `tests/unit/test_warmup.py`:
- Throughput с warmup > throughput без warmup (synthetic engine).
- Warmup time не учитывается в `elapsed`.

### 4.2 Phase 3b — BLAS thread control

**Зачем.** На гибридных Intel BLAS видит все 24 logical CPU включая
E-cores → создаёт 24 worker'ов → P-cores ждут E-cores. По
[Intel Community thread](https://community.intel.com/t5/Intel-oneAPI-Math-Kernel-Library/Weird-DGEMM-with-i9-14900k/m-p/1692021):
ограничение MKL до P-core count даёт стабильную и высокую производительность.

**Файлы:**
- `pyproject.toml`: `threadpoolctl>=3.0` в
  `[project.optional-dependencies] perf`.
- Новый `application/blas_control.py`:
  ```python
  @contextmanager
  def limit_blas_to_p_cores(system_info: SystemInfo) -> Iterator[None]:
      """Ограничить BLAS pthread-pool на P-cores только (гибридный Intel).
      Fallback chain: threadpoolctl → env-vars (deferred set) → no-op."""
  ```
  Использует `_INTEL_HYBRID_SKU_TABLE` для определения P-cores count.
  На не-hybrid CPU — yield без изменений.
- `infrastructure/stress/builtin_large_dgemm.py`: обернуть `np.matmul`-цикл
  в `with limit_blas_to_p_cores(system_info):`.

**Универсальность.**
- Hybrid Intel + threadpoolctl установлен → ограничение работает.
- Hybrid Intel + threadpoolctl нет → попытка env-vars (предупреждение
  «эффект только если выставлено ДО `import numpy`»).
- Не-hybrid x86 → no-op (BLAS уже видит правильное число cores).
- ARM / unknown → no-op.

**Тесты.** `tests/unit/test_blas_control.py`:
- SystemInfo гибридного Intel → context manager пытается ограничить.
- Ryzen → no-op без падений.
- Моки threadpoolctl для воспроизводимости.

### 4.3 Phase 3c — CPU affinity (opt-in, platform-specific)

**Зачем.** В parallel-режиме DGEMM и STREAM конкурируют за ядра. Если
прибить DGEMM к P-cores (compute-heavy), а STREAM к E-cores (memory-heavy
не страдает от слабого compute), contention падает.

**Файлы:**
- Новый `application/cpu_affinity.py`:
  ```python
  def set_thread_affinity(cpus: set[int]) -> bool:
      """Linux: os.sched_setaffinity. Windows: ctypes SetThreadAffinityMask.
      macOS: no-op. Возвращает False при ошибке/no-op."""

  def hybrid_split_for_parallel(system_info: SystemInfo) -> tuple[set[int], set[int]] | None:
      """Вернуть (p_cpu_ids, e_cpu_ids) для гибрида, иначе None."""
  ```
- `application/parallel_runner.py`: опциональный `affinity_plan: dict[int,
  set[int]] | None` — индекс engine → CPU IDs. Default None = backward-compat.
- `application/stress_orchestrator.py`: при детекции гибрида автоматически
  передавать `affinity_plan` (P-cores для DGEMM, E-cores для STREAM).

**Универсальность.**
- Linux hybrid Intel → работает.
- Windows hybrid Intel → работает (через ctypes).
- macOS / не-hybrid → no-op без падения.
- Logging при отсутствии прав (Windows может требовать admin).

**Тесты.** `tests/unit/test_cpu_affinity.py`:
- Hybrid Intel SystemInfo → возвращает split.
- Ryzen → None.
- Mock os.sched_setaffinity / win32api.

### 4.4 Phase 3d — Workload sizing audit + dynamic adjustment

**Зачем.** Сейчас:
- DGEMM N по McCalpin Rule `3·N²·8 ≥ 4·L3`. Для L3=30MB → N≈2600, матрица
  65MB. Это 2× L3, но `np.matmul` блочит на L1/L2 → cache-aware.
- STREAM per-thread buffer 120MB × 24 threads = 1.92 GB на i9-12900K. Это
  ≥60× L3, хорошо для memory-bound.

**Что проверить:**
- N в DGEMM на не-hybrid CPU тоже даёт ≥ 1 секунды на iteration.
- На server CPU с большим L3 (Xeon 38MB+) — не переходит ли N за 4096?
- На low-end CPU (i3, L3=3MB) — не падает ли N ниже 1024?

**Файлы:**
- `infrastructure/stress/builtin_large_dgemm.py`: логировать выбранный N
  + iteration time. Если iteration < 100ms → bump N до следующей multiple of 64.
- `infrastructure/stress/builtin_large_stream.py`: логировать
  suggested_size_mb + cap на `min(free_ram_gb × 0.5, 1024)`.

**Тесты.** Параметризованные на L3 от 3MB до 96MB, free_ram от 4GB до
512GB. Проверка границ `[1024, 4096]` для N.

## 6. Что НЕ делать (антипаттерны)

1. **Не заменять `np.matmul` на ручной DGEMM** (C/Cython/Numba/inline
   assembly). NumPy + BLAS — proven, cross-platform. Свой DGEMM = тысячи
   строк + поддержка SIMD по архитектурам. Williams 2009: «use optimized
   libraries first, custom kernels last».
2. **Не подгонять размеры workload под конкретный CPU.** Решение должно
   работать от i3-8100 до Xeon Platinum. Все параметры — производные от
   L3 size, cores count, free RAM.
3. **Не вводить MKL hard dependency.** Binary blob, нарушит pip install
   на не-x86 (ARM macOS, AstraLinux ARM64). Используем то что NumPy сам
   выбрал (обычно OpenBLAS на pip-wheels).
4. **Не делать NUMA-aware allocation на single-socket desktop.** Избыточная
   сложность с нулевым выигрышем. Phase 3 для single-socket; multi-socket
   — future work.
5. **Не заменять McCalpin Triad на «более эффективный» алгоритм.**
   Нарушает стандарт. Triad = `a + b·s`, точка.
6. **Не добавлять inline AVX-512 intrinsics.** Cross-arch проблемы.
   Numba/Cython могут быть рассмотрены как opt-in future work.
